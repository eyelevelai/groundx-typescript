# groundx-typescript — Contributor Guide

This repository is the **official GroundX TypeScript SDK** (`npm install groundx`). It is
the TypeScript equivalent of [`groundx-python`](https://github.com/eyelevelai/groundx-python) — both are generated from the same upstream Fern API definition by separate generators.

This file is the canonical contribution guide for agents and humans working in this
repo. Other agent-tool surfaces (Codex, Claude, Cursor, Replit, Gemini/Antigravity)
should **route here** rather than duplicate rules — see [§10](#10-other-agent-surfaces).

---

## 1. Setup, install, build, test

The repo uses **pnpm** (pinned to 10.33.0 via `packageManager`). Node 18+ supported
(also runs on Vercel, Cloudflare Workers, Deno, Bun, React Native).

```sh
# Install dependencies (CI uses --frozen-lockfile)
pnpm install

# Build (dual CJS + ESM output to dist/)
pnpm build              # = build:cjs + build:esm

# Run tests
pnpm test               # all vitest projects
pnpm test:unit          # unit project only
pnpm test:wire          # wire/mock-server (MSW) project only

# Lint + format (Biome)
pnpm check              # lint + format check
pnpm check:fix          # autofix
```

**Before opening a PR:** run `pnpm build` and `pnpm test` locally. Both must pass — they
are the two CI gates (see [§7](#7-ci-and-release-flow)).

**Note:** Biome (`pnpm check`) is available for lint/format but is **not currently a CI
gate** — only `build` (compile) and `test` are enforced in CI. Run Biome on the files
you've changed (`pnpm check`) as good hygiene, but a green build+test is what CI
requires.

## 2. Generated vs hand-written code

Most of this repo is **generated by Fern** from the upstream GroundX API definition.
The generator version and origin spec commit are pinned in `.fern/metadata.json`.

The boundary between generated and hand-written code is **`.fernignore`**, not the
directory layout. Read `.fernignore` before editing anything in `src/`. Files listed
there survive regeneration; everything else gets rewritten on the next `fern generate`.

Current `.fernignore` hand-written entries:

| Path | Why it's hand-written |
|---|---|
| `src/Wrapper.ts` | `GroundXClient extends FernClient`; adds the `ingest()` convenience method + upload helpers. The only real hand-written logic. |
| `src/index.ts` | Public export surface. |
| `README.md` | User-facing README — preserved across regenerations. |
| `openspec` | Spec-driven-development directory (see [§8](#8-spec-driven-development-openspec)). |
| `AGENTS.md` | This guide. |
| `.claude` | OpenSpec Claude Code commands + skills. |

Everything else — `Client.ts`, `BaseClient.ts`, all of `src/api/**` and `src/core/**`,
`environments.ts`, `src/auth/**`, and the generated tests — is regenerated and must not
be hand-edited.

## 3. Contributing to generated code

Generated code changes happen upstream, not here:

- **API-shape changes** (new endpoints, fields, types) — change the upstream Fern API
  definition, which lives in
  [`eyelevel-fern-config`](https://github.com/eyelevelai/eyelevel-fern-config). That
  repo holds the OpenAPI spec from which this SDK (and
  [`groundx-python`](https://github.com/eyelevelai/groundx-python)) are generated. The
  SDK is regenerated from it and re-released — do not hand-edit generated code here to
  achieve an API-shape change.
- **Generator behavior changes** (how every generated file looks — client class, error
  types, serialization, retry policy) — these come from the
  [Fern repository](https://github.com/fern-api/fern), where the TypeScript SDK
  generator lives.

For a specific file in the generated tree that genuinely needs hand-written behavior,
add it to `.fernignore` (see [§2](#2-generated-vs-hand-written-code)) and write it by
hand. Use this sparingly — every entry is a maintenance commitment.

To coordinate an upstream API change: external contributors should open a
[GitHub issue](https://github.com/eyelevelai/groundx-typescript/issues) describing the
needed shape change; a maintainer routes it to the
[`eyelevel-fern-config`](https://github.com/eyelevelai/eyelevel-fern-config) repo.
Internal contributors should track the change in Linear.

## 4. The hand-written layer (`src/Wrapper.ts`)

`src/Wrapper.ts` is the entire editable logic surface (~282 lines). Its public API is
exactly one method:

```ts
ingest(documents, callbackUrl?, callbackData?, requestOptions?, uploadApi?)
```

It partitions documents into remote (URL) and local (filesystem path), uploads local
files to the presigned upload service, then calls the generated
`documents.ingestRemote(...)`. Private helpers handle presigned-URL fetch, the byte
upload, file-type derivation, and custom auth headers.

**Naming:** `GroundXClient` is the class users instantiate (hand-written, extends the
generated `FernClient`); `GroundX` is the types namespace.

**Divergence from the Python twin.** `groundx-python`'s hand-written layer is much
larger — it adds batching, directory ingest, CSV splitting, an `extract` subsystem, and
async variants. **This SDK has none of those** — a single, unbatched, serial `ingest()`.
Those are feature-parity candidates if ever ported, not current behavior.

---

## 5. Extending the hand-written layer

- **Add logic to `ingest()` or a new helper** — edit `src/Wrapper.ts` directly (it's
  already on `.fernignore`). Export any new public surface via `src/index.ts`.
- **Add a brand-new hand-written file** — create it, then **add its path to
  `.fernignore` first** or Fern will clobber it on the next regen. Export via
  `src/index.ts`.
- **Never** rely on the generated `src/index.ts` top-level exports knowing about new
  hand-written symbols — wire them through the hand-written export surface.

## 6. Credentials and tests

CI tests **do not use live GroundX credentials.** Unit and wire tests run against mocked
HTTP (MSW). A contributor can clone and run the full suite without a GroundX account.

- **Never commit credentials.** Read the API key from the environment in any local
  scripts; gate any live test behind an env var so it skips when the key is absent.
- **Do not add a CI job that requires shared secrets** without coordinating with
  maintainers.

## 7. CI and release flow

CI (`.github/workflows/ci.yml`) runs on every push:

- **`compile`** job → `pnpm build` (must pass)
- **`test`** job → `pnpm test` (must pass)
- **`publish`** job → needs both; runs only on tag pushes, publishes to npm.

These two — **build and test** — are the gates an agent must satisfy before a PR can
merge. Releases are triggered by tag pushes (alpha/beta/backport tag logic in CI).

## 8. Spec-driven development (OpenSpec)

Non-trivial changes to the hand-written layer start as an **OpenSpec** proposal, not as
code. The spec is the durable record of *what* is changing and *why*, reviewed before
implementation.

- **Where specs live:** `openspec/changes/` (active proposals),
  `openspec/changes/archive/` (completed), `openspec/specs/` (current canonical specs).
- **Workflow (Claude Code slash commands):** `/opsx:explore` to investigate →
  `/opsx:propose` to draft a change → review the generated proposal/specs/tasks →
  `/opsx:apply` to implement (test-first) → `/opsx:sync` / `/opsx:archive` once merged.
  Run `openspec` (the CLI) for the current command set if these differ.
- `openspec/` is listed in `.fernignore`, so Fern regeneration never touches it.
- **Scope:** OpenSpec changes in this repo target **only** the hand-written `.fernignore`
  layer (`src/Wrapper.ts`, `src/index.ts`). API-surface changes belong upstream in
  [`eyelevel-fern-config`](https://github.com/eyelevelai/eyelevel-fern-config) (see
  [§3](#3-contributing-to-generated-code)), not here.

## 9. Workflow

1. For a non-trivial change, propose an OpenSpec change first (see
   [§8](#8-spec-driven-development-openspec)).
2. Branch from `main` with a descriptive name.
3. Make changes scoped to the hand-written layer ([§4](#4-the-hand-written-layer-srcwrapperts)
   / [§5](#5-extending-the-hand-written-layer)); never edit generated code.
4. Run local gates before pushing — `pnpm build` and `pnpm test`. Both must pass.
5. Open a PR against `main`. CI runs `compile` (build) and `test`. Both must pass.
6. After merge, a maintainer tags a release (see [§7](#7-ci-and-release-flow)) to ship to npm.

**Commit messages.** Clear, descriptive, present tense. Reference the relevant Linear
ticket ID when applicable.

**Code style.** Biome (`pnpm check` / `pnpm check:fix`). Not a CI gate, but run it on the
files you've changed.

## 10. Other agent surfaces

**Do not duplicate this file** into agent-specific variants (`CLAUDE.md`, `CURSOR.md`,
`.cursorrules`, etc.). Duplicated rules drift, then disagree, then mislead.

Acceptable: a one-line agent-specific file that reads simply `See @AGENTS.md`. Anything
more than that pointer goes here. (Note: the `.claude/` directory holds OpenSpec
workflow commands and skills generated by `openspec init` — those are tool wiring, not a
parallel rulebook, and are fine.)

## 11. Boundaries

- ✅ **Always do:** run `pnpm build` and `pnpm test` before pushing, read credentials
  from the environment, edit code only in paths listed in `.fernignore`
  (`src/Wrapper.ts`, `src/index.ts`)
- ⚠️ **Ask first** (open a GitHub issue / Linear ticket): add a path to `.fernignore`,
  change `.fern/metadata.json` or `ci.yml` non-trivially, add a CI job requiring shared
  credentials
- 🚫 **Never do:** edit Fern-generated files directly (overwritten on next regen — this
  includes `Client.ts`, `BaseClient.ts`, `src/api/**`, `src/core/**`, `environments.ts`,
  `src/auth/**`), achieve an API-shape change by hand-editing here instead of upstream in
  `eyelevel-fern-config`, remove a path from `.fernignore` without first re-homing the
  hand-written code, commit credentials in any form, create agent-tool-specific files
  that duplicate this one

## 12. Known gaps (current)

These are real, verified issues in the hand-written layer — useful context for any
change, and candidate work items:

- **Zero test coverage on `ingest()`.** No test imports `Wrapper.ts` or calls `ingest()`;
  the wire tests exercise the generated client, and `tests/custom.test.ts` is a
  placeholder. CI will not catch a regression in the hand-written layer.
- **The byte-upload PUT is unguarded.** The presigned-URL upload uses a raw global
  `fetch` with no retries/timeout/auth-provider/logging — it bypasses the shared core
  fetch pipeline that every generated call inherits.
- **Two parallel auth paths.** The generated client uses the `AuthProvider`; the wrapper
  upload uses its own `_getCustomAuthorizationHeaders()`. Both emit `X-API-Key`; a change
  to auth must touch both.
- **No response validation.** The generator runs with `noSerdeLayer` +
  `skipResponseValidation`, so success bodies are cast, never schema-checked.
